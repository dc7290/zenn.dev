# はじめに

React コンポーネントの設計は、規模が大きくなるにつれて難しくなっていきます。boolean props が増えていったり、`renderHeader` のような render props が乱立したり、コンポーネントの中身が条件分岐だらけになったり。

こうした問題へのアプローチとして **Composition Patterns** という考え方があります。この記事では Vercel が公開している [vercel-composition-patterns](https://skills.sh/vercel-labs/agent-skills/vercel-composition-patterns) を参照しつつ、その中でも **Compound Components** パターンに焦点を当てて、実際のコード例とともに紹介させていただきます。

また、Composition Patterns と Monolithic なコンポーネント設計のそれぞれにメリットがあることについても触れていきます。

:::message
この記事のコード例では説明の簡略化のため import 文やスタイルの記述を省略している場合があります。
:::

# Compound Components とは

**Compound Components**（複合コンポーネント）は、1 つの機能を複数のサブコンポーネントに分割し、使用者側でそれらを組み合わせて使うパターンです。

HTML の `<select>` と `<option>` の関係がわかりやすい例です。`<select>` が単体で `items` のような属性を受け取ってリストを描画するのではなく、`<option>` を子要素として組み合わせて構成します。

```html
<!-- Configuration的: select が items を受け取る（実際のHTMLはこうではない） -->
<select items="[{value: 'a', label: 'A'}, {value: 'b', label: 'B'}]" />

<!-- Composition的: 実際のHTMLの姿 -->
<select>
  <option value="a">A</option>
  <option value="b">B</option>
</select>
```

React における Compound Components もこれと同じ考え方です。具体例を見てみます。

## Monolithic なコンポーネント

```tsx
function Composer({
  onSubmit,
  isThread,
  channelId,
  isDMThread,
  dmId,
  isEditing,
  isForwarding,
}: Props) {
  return (
    <form>
      <Header />
      <Input />
      {isDMThread ? (
        <AlsoSendToDMField id={dmId} />
      ) : isThread ? (
        <AlsoSendToChannelField id={channelId} />
      ) : null}
      {isEditing ? (
        <EditActions />
      ) : isForwarding ? (
        <ForwardActions />
      ) : (
        <DefaultActions />
      )}
      <Footer onSubmit={onSubmit} />
    </form>
  );
}
```

boolean props による分岐が内部に隠蔽されています。props を追加するたびに可能な状態の組み合わせは指数的に増えていきます。

## Compound Components で書き直す

```tsx
function ChannelComposer() {
  return (
    <Composer.Frame>
      <Composer.Header />
      <Composer.Input />
      <Composer.Footer>
        <Composer.Attachments />
        <Composer.Formatting />
        <Composer.Emojis />
        <Composer.Submit />
      </Composer.Footer>
    </Composer.Frame>
  );
}

function ThreadComposer({ channelId }: { channelId: string }) {
  return (
    <Composer.Frame>
      <Composer.Header />
      <Composer.Input />
      <AlsoSendToChannelField id={channelId} />
      <Composer.Footer>
        <Composer.Formatting />
        <Composer.Emojis />
        <Composer.Submit />
      </Composer.Footer>
    </Composer.Frame>
  );
}

function EditComposer() {
  return (
    <Composer.Frame>
      <Composer.Input />
      <Composer.Footer>
        <Composer.Formatting />
        <Composer.Emojis />
        <Composer.CancelEdit />
        <Composer.SaveEdit />
      </Composer.Footer>
    </Composer.Frame>
  );
}
```

それぞれのバリエーションが何をレンダリングするか一目瞭然で、状態数も加算的（3 バリアント = 3 状態）になります。vercel-composition-patterns では、こうした明示的なバリアントコンポーネントを **Explicit Variants** と呼んでいます。

# Headless UI ライブラリと Compound Components

近年の主要な Headless UI ライブラリは、いずれも Compound Components で設計されています。

- **Radix UI** — `Dialog.Root` / `Dialog.Trigger` / `Dialog.Content`
- **Base UI** — `Select.Root` / `Select.Trigger` / `Select.Popup`
- **Ark UI** — `Menu.Root` / `Menu.Trigger` / `Menu.Content`

例えば Radix UI の `DropdownMenu` は以下のように使います。

```tsx
<DropdownMenu.Root>
  <DropdownMenu.Trigger>メニューを開く</DropdownMenu.Trigger>
  <DropdownMenu.Portal>
    <DropdownMenu.Content>
      <DropdownMenu.Item>編集</DropdownMenu.Item>
      <DropdownMenu.Item>削除</DropdownMenu.Item>
    </DropdownMenu.Content>
  </DropdownMenu.Portal>
</DropdownMenu.Root>
```

これらのライブラリが Compound Components を採用しているのは、使用者が `children` を通じて自由に構成できるようにするためです。これは裏を返すと、**ライブラリをラップするアプリケーション側のコンポーネントも、この思想に沿って設計しないとライブラリの仕組みを活かしきれない** ということを意味しています。

# アプリケーション側のコンポーネント設計をライブラリに合わせる

具体例で見ていきます。Radix UI の `DropdownMenu` をラップしてドロップダウンメニューコンポーネントを作るとき、2 つのアプローチが考えられます。

## アプローチ A: children ベース

```tsx
const DropdownMenuContent = ({ children, ...props }) => {
  return (
    <DropdownMenu.Portal>
      <DropdownMenu.Content {...props}>{children}</DropdownMenu.Content>
    </DropdownMenu.Portal>
  );
};
```

使用者側は `DropdownMenuItem` などを自由に組み合わせます。

```tsx
<DropdownMenu>
  <DropdownMenuTrigger>メニュー</DropdownMenuTrigger>
  <DropdownMenuContent>
    <DropdownMenuItem onSelect={handleEdit}>編集</DropdownMenuItem>
    <DropdownMenuItem onSelect={handleDelete}>削除</DropdownMenuItem>
  </DropdownMenuContent>
</DropdownMenu>
```

## アプローチ B: items 配列 props

```tsx
const DropdownMenuContent = ({ items, ...props }) => {
  return (
    <DropdownMenu.Portal>
      <DropdownMenu.Content {...props}>
        {items.map((item, i) => (
          <DropdownMenu.Item key={item.label} onSelect={item.onSelect}>
            {item.label}
          </DropdownMenu.Item>
        ))}
      </DropdownMenu.Content>
    </DropdownMenu.Portal>
  );
};
```

使用者側は `items` 配列でメニューを定義します。

```tsx
<DropdownMenu>
  <DropdownMenuTrigger>メニュー</DropdownMenuTrigger>
  <DropdownMenuContent
    items={[
      { label: "編集", onSelect: handleEdit },
      { label: "削除", onSelect: handleDelete },
    ]}
  />
</DropdownMenu>
```

一見するとアプローチ B でも十分に見えます。しかし、**メニュー項目をクリックしたときに確認ダイアログ（AlertDialog）を開きたい** という要件が出てきたときに大きな差が生まれます。

## Compound Components なら自然に組み合わせられる

アプローチ A の場合、Radix UI の `AlertDialog` をそのまま組み合わせることができます。

```tsx
<DropdownMenu>
  <DropdownMenuTrigger>メニュー</DropdownMenuTrigger>
  <DropdownMenuContent>
    <DropdownMenuItem onSelect={handleEdit}>編集</DropdownMenuItem>
    <AlertDialog>
      <AlertDialogTrigger asChild>
        <DropdownMenuItem
          onSelect={(e) => e.preventDefault()} // DropdownMenuContentが閉じるのを一時的にキャンセルして、AlertDialogがDOMから消えるのを防ぐ必要がある
        >
          削除
        </DropdownMenuItem>
      </AlertDialogTrigger>
      <AlertDialogContent>
        <AlertDialogTitle>本当に削除しますか？</AlertDialogTitle>
        <AlertDialogDescription>
          この操作は取り消せません。
        </AlertDialogDescription>
        <AlertDialogActionGroup>
          <AlertDialogAction onClick={handleDelete}>削除</AlertDialogAction>
          <AlertDialogCancel />
        </AlertDialogActionGroup>
      </AlertDialogContent>
    </AlertDialog>
  </DropdownMenuContent>
</DropdownMenu>
```

Radix UI の `asChild` は子要素に props をマージする仕組み（Slot パターン）を持っています。`<AlertDialogTrigger asChild>` で `<DropdownMenuItem>` をラップすることで、メニュー項目のクリックがそのままダイアログのトリガーになります。

ライブラリ側もアプリケーション側も Compound Components で設計されているからこそ、**異なるコンポーネント同士をレゴブロックのように組み合わせられる** わけです。

## この設計のままだと組み合わせられない

アプローチ B のように `label` / `icon` / `onSelect` だけを受け取る閉じた `items` API にすると、各項目の中に `AlertDialog` のようなコンポーネントツリーを差し込めません。結果として以下のような実装になります。

```tsx
const [isAlertOpen, setIsAlertOpen] = useState(false);

return (
  <>
    <DropdownMenu>
      <DropdownMenuTrigger>メニュー</DropdownMenuTrigger>
      <DropdownMenuContent
        items={[
          { label: "編集", icon: "edit", onSelect: handleEdit },
          {
            label: "削除",
            icon: "delete",
            onSelect: () => setIsAlertOpen(true),
          },
        ]}
      />
    </DropdownMenu>

    {/* DropdownMenu の外に AlertDialog を配置し、open state で手動管理 */}
    <AlertDialog open={isAlertOpen} onOpenChange={setIsAlertOpen}>
      <AlertDialogContent>
        <AlertDialogTitle>本当に削除しますか？</AlertDialogTitle>
        <AlertDialogDescription>
          この操作は取り消せません。
        </AlertDialogDescription>
        <AlertDialogActionGroup>
          <AlertDialogAction onClick={handleDelete}>削除</AlertDialogAction>
          <AlertDialogCancel />
        </AlertDialogActionGroup>
      </AlertDialogContent>
    </AlertDialog>
  </>
);
```

`AlertDialogTrigger` が使えないため `useState` で `open` 状態を手動管理し、コンポーネントのツリー上も DropdownMenu とは別の場所に AlertDialog を配置するしかありません。

このように設計すると Radix UI が提供する `Trigger` / `asChild` といった仕組みが活かせなくなります。
これは単純にコードヘルスとして不利なだけでなく、`AlertDialogTrigger` を使わないため `AlertDialogCancel` を押したときにトリガーへフォーカスが戻らない、というユーザー影響がある問題も発生します。

:::details アプローチBでitemsの方を拡張する代替案
items 配列で、React.ReactNodeやrender propを受け取る方法でAlertDialogTriggerを使用する方法も考えられますが、そのように対応していくうちにどんどんitemsの型が膨らんでいき、管理が難しくなっていくという問題もあるので、アプローチAを取る方がシンプルで良いかなと思います。
:::

## Compound Components による合わせ方の別の例：フォーム

DropdownMenu 以外にも、Compound Components でライブラリに合わせた設計が有効なケースがあります。

フォーム要素を考えます。ラベル・説明文・エラーメッセージ・文字数カウントなどの周辺要素と、Input や Select などのフォーム要素を組み合わせる必要があります。

Monolithic に設計すると以下のようになります。

```tsx
<InputField
  label="タイトル"
  description="記事のタイトルを入力してください"
  required
  error={errors.title}
  value={title}
  onChange={setTitle}
/>
```

一見整理されていますが、InputField が多くの責務を持ち、対応するフォーム要素が増えるたびに `InputField` 側の考慮が増えていきます。ラベルとフォーム要素の横並び、ラベルなしの場合、FieldSet でのグルーピングなど、バリエーションに対応するために props が膨らんでいきます。

Compound Components で設計すると以下のようになります。

```tsx
<Field required>
  <FieldLabel>タイトル</FieldLabel>
  <FieldDescription>記事のタイトルを入力してください</FieldDescription>
  <Input value={title} onChange={setTitle} />
  <FieldError error={errors.title} />
</Field>
```

`Field` が `Provider` で Context を提供し、`FieldLabel` や `FieldError` が Context 経由でアクセシビリティ属性（`aria-labelledby` / `aria-describedby`）を自動的に紐付けます。フォーム要素側は Context を通じてこの仕組みと統合します。

この設計であれば、新しいフォーム要素が追加されても `Field` 側の変更は不要です。

:::message
BaseUIはまさにこの設計を採用しています。
:::

# Monolithic パターンのメリット

ここまで Compound Components の威力を紹介してきましたが、Monolithic な設計にも明確なメリットがあります。

## 型による制約の強制

Monolithic なコンポーネントでは **TypeScript の型システムがそのまま使い方のルールとして機能** します。

```tsx
type DialogProps = {
  title: string; // 必須 — 渡さないとコンパイルエラー
  description?: string;
  children: React.ReactNode;
  onConfirm: () => void;
};
```

Compound Components では `children` で自由に構成できるため、必須のサブコンポーネントやアクセシビリティ要件が満たされているかを型で保証できません。

```tsx
<AlertDialog>
  <AlertDialogContent>
    {/* accessible name を与える手段を入れ忘れた！でもコンパイルは通る */}
    <AlertDialogDescription>本当に削除しますか？</AlertDialogDescription>
    <AlertDialogActionGroup>
      <AlertDialogAction>削除</AlertDialogAction>
      <AlertDialogCancel />
    </AlertDialogActionGroup>
  </AlertDialogContent>
</AlertDialog>
```

Radix UI ではこのようなケースで開発時に `console.warn` が出力されますが、ビルド時には検出できません。Compound Components で同等の制約を設けるには、ランタイム警告や ESLint カスタムプラグインなどの代替手段が必要で、props の `title: string` を必須にする 1 行と比較するとコストが段違いです。

:::message
`alertdialog` ロールは、`aria-labelledby` か `aria-label` を通じてアクセス可能な名前を持つ必要があります。`AlertDialogTitle` はそのための代表的な手段ですが、`AlertDialogContent` に `aria-label` や `aria-labelledby` を与えて要件を満たすこともできます。
:::

## 使い方の統一

Compound Components は使用者に自由を与える反面、設計者が意図しない使い方ができてしまいます。

```tsx
<DropdownMenuContent>
  {/* DropdownMenuItem ではなく任意の要素を入れてしまう */}
  <div onClick={handleClick}>カスタム要素</div>
</DropdownMenuContent>
```

コンポーネントの使い方をある程度決め打ちして、プロダクト全体の UI の一貫性を保ちたい場合には、Monolithic な設計も有効と考えます。

# ハイブリッドという選択肢

実際にmicroCMSのコンポーネント設計では、Compound Components と Monolithic の**ハイブリッド**な設計を採用しているケースがあります。

例えば `DropdownMenu` は、Compound Components と Monolithic のハイブリッドな設計を採用しており、`DropdownMenuItem` は Monolithic な設計を採用しています。

```tsx
const DropdownMenuItem = forwardRef<
  HTMLDivElement,
  Omit<DropdownMenu.DropdownMenuItemProps, "children"> & {
    label: string;
    description?: React.ReactNode;
    icon?: string;
  }
>(({ className, label, description, icon, ...props }, forwardedRef) => {
  return (
    <DropdownMenu.Item
      ref={forwardedRef}
      className={cx(styles.item, className)}
      {...props}
    >
      {icon && (
        <Icon
          name={icon}
          size={description ? "large" : "small"}
          className={styles.icon}
        />
      )}
      <span className={styles.textGroup}>
        <span className={styles.label}>{label}</span>
        {description && (
          <span className={styles.description}>{description}</span>
        )}
      </span>
    </DropdownMenu.Item>
  );
});
```

`DropdownMenuContent` 内は `DropdownMenuGroup` や `DropdownMenuSeparator` などのコンポーネントを組み合わせ可能にするため、Compound Components として設計されていますが、上記のように `DropdownMenuItem` は Monolithic な設計を採用しています。
`DropdownMenuItem` 内は、DOM構造であったりdescriptionの有無によってiconサイズを固定化したかったのもありコンポーネント側で強制しています。

「このコンポーネントは Compound と Monolithic の間でどのラインが最適なのか」を判断するのがエンジニアの腕の見せ所なのかなと思います（笑）。

# まとめ

この記事では [vercel-composition-patterns](https://skills.sh/vercel-labs/agent-skills/vercel-composition-patterns) を参照しつつ、Compound Components パターンについて実際のコード例とともに紹介させていただきました。

- **Compound Components で柔軟に構成する**: 使用者側が `children` で自由にサブコンポーネントを組み合わせる
- **ライブラリの設計思想と合わせる**: Radix UI のような Compound Components で設計されたライブラリをラップする場合、アプリケーション側も同じ思想に沿うことで `asChild` や `Trigger` の仕組みをフルに活かせる
- **Monolithic にもメリットがある**: 型による制約の強制や使い方の統一など、Monolithic が適しているケースもある

コンポーネントの設計に唯一の正解はありませんが、ライブラリが Compound Components で提供されている今、その思想を理解した上で使い分けることが重要だと考えています。
